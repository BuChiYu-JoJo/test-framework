## 一、项目背景与目标

### 1.1 背景

AI 时代开发周期缩短，测试平台需要同时满足以下要求：

- 用例编写门槛低，不懂代码的人也能参与
- 新项目适配快，减少重复建设成本
- 复杂场景可通过扩展方式覆盖
- 平台本身具备稳定的资产沉淀和扩展能力

### 1.2 目标

1. **降低门槛**  
   非技术人员可通过 Excel / Web 界面编写基础用例。
2. **提升效率**  
   一次配置，多项目复用，支持自动执行与实时反馈。
3. **保障质量**  
   简单场景通过配置驱动，复杂场景允许代码扩展。
4. **可持续演进**  
   以当前 UI 自动化平台为基线，逐步扩展到接口、性能、SEO、黑白盒测试等场景。

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| 配置驱动 | 元素定位、项目配置、用例内容尽量配置化，不硬编码 |
| 分层隔离 | API 层、业务层、执行引擎层职责分离 |
| 兼容演进 | 在不破坏当前主链路的前提下扩展新模块 |
| 渐进复杂 | Excel/YAML 解决简单场景，代码扩展解决复杂场景 |
| 真实基线 | 现状设计以当前源码为准，目标设计明确标记为规划态 |

## 四、用例编写规范

### 4.1 用例格式分层

当前和规划设计仍保持原始分层思想，但以当前真实实现范围为准：

| 层级 | 格式 | 适用人群 | 当前状态 |
|------|------|---------|--------|
| L1 | Excel 表格 | 不懂代码的测试人员 | 已实现（导入） |
| L2 | YAML 声明式 | 测试工程师 | 已实现 |
| L3 | Python 关键字扩展 | 测试开发 | 规划保留，未形成正式平台能力 |
| L4 | Python 纯代码 | 高复杂场景 | 规划保留，未形成正式平台能力 |

### 4.2 L1: Excel 格式（零门槛）

设计目标：

- 用于批量录入和低门槛维护测试步骤
- 通过导入器转换为标准 YAML 内容

示例结构：

```text
| 用例ID | 用例名称 | 模块 | 优先级 | 操作类型 | 目标元素 | 输入值 | 预期结果 | 备注 |
|--------|---------|------|--------|---------|---------|--------|---------|------|
| TC001  | 登录成功 | login | P0 | navigate | - | /login | 进入登录页 | |
| TC001  | 登录成功 | login | P0 | type | username | demo | 输入成功 | |
| TC001  | 登录成功 | login | P0 | click | submit | - | 点击成功 | |
```

当前实现说明：

- 支持 `.xlsx/.xls`
- 支持一组多行合成一条用例
- 导入结果统一落为 YAML 文本

### 4.3 L2: YAML 格式

当前执行引擎正式支持 YAML 作为标准执行载体。

```yaml
id: TC001
name: 登录-正确账号密码
module: login
priority: P0
tags: [smoke, core]
version: 1.0.0
author: tester001

steps:
  - no: 1
    action: navigate
    target: /dashboard/login/

  - no: 2
    action: type
    target: login.username
    value: demo

  - no: 3
    action: click
    target: login.submit
```

当前实现说明：

- `engine/parser/yaml_parser.py` 负责解析和标准化
- 引擎运行时会把 `Case.content` 写成临时 YAML 文件再执行

### 4.4 L3: Python 关键字

设计保留：

- 允许未来通过 Python 关键字封装复杂公共流程
- 当前仓库尚未形成完整平台化的关键字插件体系

示例形态：

```python
@keyword("登录系统")
def login_system(username: str, password: str):
    ...
```

### 4.5 L4: Python 纯代码（复杂场景）

设计保留：

- 用于未来覆盖高复杂度测试场景
- 当前产品基线不将其视为平台正式能力

### 4.6 当前支持的 YAML 关键字段

| 字段 | 说明 |
|------|------|
| id | 用例编号 |
| name | 用例名称 |
| module | 模块名 |
| priority | 优先级 |
| tags | 标签 |
| version | 版本 |
| author | 作者 |
| setup | 前置步骤 |
| steps | 主步骤 |
| teardown | 后置步骤 |
| expected | 预期结果 |
| data | 测试数据 |

## 五、元素定位管理

### 5.1 Locators 结构

当前项目目录中仍保留 `locators.json` 作为引擎运行时输入格式，设计上采用数据库管理 + 文件兼容。

示例：

```json
{
  "version": "1.0.0",
  "pages": {
    "login": {
      "name": "登录页",
      "elements": {
        "username": {
          "selector": "[name='account']",
          "type": "css",
          "priority": 1,
          "description": "用户名输入框"
        }
      }
    }
  }
}
```

### 5.2 定位优先级规则

文档保留定位优先级思路，但当前实际实现只保证基础 selector 维护能力，不对所有类型做正式承诺。

建议优先级：

| 优先级 | 类型 | 说明 |
|--------|------|------|
| P1 | 稳定属性选择器 | 如 data-testid、id |
| P2 | 业务文本选择器 | 如按钮文本 |
| P3 | 普通 CSS 路径 | 作为兼容方案 |
| P4 | XPath | 最后手段 |

### 5.3 版本管理

当前实际情况：

- 模型中存在 `version` 字段
- 当前 API 未提供独立版本历史查询
- 当前未形成完整回滚能力

因此设计约束为：

- 版本管理为保留扩展位
- 当前不将“版本历史”视为已落地功能

### 5.4 当前 DB -> File 兼容策略

```text
Locators 页面维护数据库
-> 执行前同步为 projects/<project>/locators.json
-> 引擎通过 locator_resolver.py 读取文件
```

该策略是当前系统的重要兼容设计，不可在新设计中忽略。

## 六、测试数据管理

### 6.1 数据分类

当前项目结构中测试数据主要分为：

- 项目环境配置：`projects/<project>/env/*.json`
- 业务测试数据：`projects/<project>/test_data/*`
- 用例内嵌数据：YAML `data` 字段
- 通知与系统配置：后端 `.env.local`

### 6.2 数据隔离策略

| 策略 | 说明 | 当前状态 |
|------|------|---------|
| 项目隔离 | 每个项目使用独立目录 | 已实现 |

| 用例数据内嵌 | YAML 中定义局部数据 | 已实现 |
| 随机化隔离 | 时间戳/随机串避免污染 | 依赖具体用例，自主约定 |

### 6.3 当前限制

- 平台尚未提供统一的测试数据管理页面
- 敏感数据加密体系仍不完整
- 数据模板管理更多依赖项目目录约定

## 七、执行引擎

### 7.1 当前执行流程

```text
用户触发执行
     │
     ▼
创建 Execution 记录
     │
     ▼
后台线程启动 run_test_case
     │
     ▼
读取 Case.content 并写入临时 YAML 文件
     │
     ▼
TestEngine 解析并执行步骤
     │
     ├── 步骤结果落库 ExecutionStep
     ├── SSE 推送步骤流
     ├── 调试日志推送
     └── 截图保存
     ▼
Reporter 生成 HTML 报告
     ▼
更新 Execution 最终状态
     ▼
发送飞书通知
```

### 7.2 当前关键字库

当前 `engine/keyword_executor.py` 已具备的关键能力主要包括：

| 类别 | 关键字示例 |
|------|------------|
| 页面导航 | navigate / go / open |
| 点击操作 | click / dblclick / rightclick |
| 输入操作 | type / fill / input / clear |
| 选择操作 | select / check / uncheck |
| 等待操作 | wait / wait_for / wait_for_url / wait_for_element |
| 断言操作 | assert_text / assert_visible / assert_hidden / assert_count / assert_url / assert_exists |
| API 操作 | api_request |
| 其他操作 | screenshot / refresh / back / forward / scroll / hover / execute_js / upload / clear_cookies |

### 7.3 当前失败处理机制

当前基线能力：

- 步骤失败后整体执行可进入失败态
- 执行异常会写入 `error_msg`
- 通知失败不阻断主链路

当前未形成完整平台机制：

- 通用失败重试策略
- Flaky 自动标记
- 断点续跑

### 7.4 引擎与定位符集成

- 引擎通过 `locator_resolver.py` 读取 `locators.json`
- 执行前通过 `engine.py` 将 DB 定位符同步为文件
- 这是当前系统不可绕过的兼容层

## 十三、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 现状与目标混写导致评审失真 | 研发和评审口径混乱 | 文档严格区分基线与规划 |
